## 2024-03-20 - [Optimize Text Tokenization for Set Operations]
**Learning:** When calculating Jaccard similarity across large combinations of strings, converting memoized tuple results into sets on every comparison call incurs significant O(N) allocation overhead. Caching the output as a `frozenset` instead bypasses this conversion cost. However, `frozenset` does not preserve order, which breaks sequence-dependent functions like `difflib.SequenceMatcher`.
**Action:** When memoizing string tokenization intended for both set operations (like Jaccard) and sequence operations, create two distinct cached functions (e.g., one returning `frozenset` and one returning `tuple`) to maximize performance without sacrificing deterministic ordering where needed.

## 2025-03-09 - [Optimize Polars Fallback Chain and Replace to_dicts safely]
**Learning:** When trying to eliminate Polars FFI overhead, replacing `to_dicts()` with `zip()` iterations over column Series is incredibly fast. However, if the column contains arrays/lists, iterating using `zip` yields Polars `Series` objects instead of Python lists. Calling Python's native truth value operations (e.g. `if series_var:`) on these results in `TypeError: the truth value of a Series is ambiguous`. Furthermore, when avoiding `map_elements` as a fallback for dictionary mappings (`replace_strict`), converting the dictionary to a DataFrame and using a `.join(how="left")` is much safer and performant across older versions of Polars than defaulting back to `map_elements`.
**Action:** When replacing `to_dicts()` with `zip(...)`, always cast list-like Series objects explicitly to python lists using `.to_list()` (e.g., `series.to_list() if hasattr(series, 'to_list') else series`) before applying native Python boolean/truth logic. For `replace_strict` fallbacks, construct a DataFrame and use left join instead of `map_elements`.

## 2024-03-22 - [Optimization: Precompiled Regexes for Text Normalization]
**Learning:** In text-heavy data processing pipelines (like `produto_runtime.py` and `produto_classification.py`), inline regex compilation inside frequently called functions (like `re.sub(r"[^A-Z0-9 ]+", " ", text)`) adds substantial overhead. Furthermore, using `re.sub(r"\s+", " ", text)` to collapse multiple spaces is measurably slower than the native string operation `' '.join(text.split())`.
**Action:** Always precompile regex patterns globally (`_RE_NON_ALNUM = re.compile(r"[^A-Z0-9 ]+")`) when used in iterative processing loops or `@lru_cache` functions. Prefer `' '.join(text.split())` over regex for whitespace collapsing.

## 2024-03-01 - Fast dict construction and iteration over Polars DataFrames
**Learning:** Calling `df.to_dicts()` to iterate over large DataFrames row-by-row allocates massive Python dictionaries that cross the Rust-Python FFI boundary one by one. This causes huge iteration overhead when the data frames grow. In `server/python/core/produto_batch_lote.py`, we iterate over `df_agregados` and `df_pairs`, reconstructing dictionaries to execute complex comparison logic. Using `zip()` over `Series.to_list()` runs faster. A pattern of `[dict(zip(cols, vals)) for vals in zip(*extracts)]` performs better than `to_dicts()` because Python tuple unpacking and `dict()` natively zip in C, avoiding repetitive FFI overhead inside the loop.
**Action:** Replace `df.to_dicts()` calls with the fast `zip()` and column extraction method in performance-critical areas, or use native Polars expressions when possible.
